Crafting Good Commit Messages: A Developer’s Guide

As developers, we know that version control is essential, but writing meaningful commit messages often gets overlooked. A well-written commit message not only helps your future self but also your team in understanding the changes and reasoning behind them. Let’s break down some best practices for crafting good commit messages.

  1. Keep it Short, but Informative

A commit message should give a quick overview of what the commit does. The general rule is that the summary line should be no more than 50 characters. This makes it readable in git logs and on platforms like GitHub.

❌ "fixes bug"
✅ "Fix issue where user session was not cleared on logout"
  1. Use the Imperative Mood

Write your commit messages in the imperative mood, as if you’re giving a command. Think of it like this: when you run git log, you are telling Git what the commit does.

❌ "fixed the login bug"
✅ "Fix login bug when session times out"

The imperative mood emphasizes the action being done and reads well in lists of commits. 3. Explain Why, Not Just What

Sometimes, a simple description of what was changed isn’t enough. Include a brief explanation of why the change was made, especially if the reasoning isn’t obvious.

❌ "Remove deprecated method"
✅ "Remove deprecated method to prevent security vulnerabilities"

This adds context for anyone reviewing the commit later. 4. Break Longer Descriptions into a Body

If your change requires more explanation, add a detailed description in the body of the commit message. Leave a blank line after the summary and then provide more details.

Example:

vbnet

Fix bug where user session wasn’t cleared on logout

This bug caused users to remain logged in after clicking “Logout”. Clearing the session data resolves the issue and improves security.

  1. Reference Relevant Issues or Tickets

Link your commits to issues or tickets by including the issue number. This adds traceability and helps developers see the full history of the problem.

✅ "Add error handling for API timeouts (#123)"

Most tools like GitHub, GitLab, or Jira will automatically create a link to the issue when referenced in the commit. 6. Commit Small, Incremental Changes

Each commit should represent a single, logical change. Avoid grouping unrelated changes in the same commit. This makes it easier to understand and roll back specific changes if needed.

❌ "Fix bugs and add new features"
✅ "Fix bug in logout logic"
✅ "Add user role management feature"
  1. Capitalize and Punctuate Correctly

Start your commit messages with a capital letter and generally avoid punctuation at the end of the subject line. This keeps it clean and consistent.
